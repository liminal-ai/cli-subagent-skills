  What Codex Has Been Tightening

  The modified files tell a consistent story — Codex is converging all four
  skills on a unified --json exec pattern:

  1. Smoke tests (claude, copilot, cursor) — now all do a single --json exec
  call and destructure both .result and .session_id from the JSON response.
  Cursor adds a perl whitespace trim for its output.
  2. copilot-result — gained --json flag for exec mode, session-file mode, and
  fallback session ID resolution (tries session.start event, falls back to
  result event).
  3. cursor-result — bigger rewrite: --json flag, stream JSONL saving alongside
  Cursor's transcript, stream-file mode, richer tool/command extraction from
  stream events.
  4. SKILL.md files — all updated to recommend explicit session ID capture from
  --json exec for automation, rather than resolving the latest session by cwd.

  Key Conventions to Know

  - {name}-result --json exec "prompt" → returns {session_id, result, model, 
  usage, session_file}
  - Codex is the one source that's still stateless (reads arbitrary JSONL files
  you manage); the other three resolve sessions by working directory
  - Release artifacts (dist/) are gitignored — both .zip and .skill files must
  be manually uploaded via gh release create/upload

  Claude-subagent

  Overall: Works well. Most mature. A few rough edges worth documenting.

  What worked:
  - Sync exec clean and fast (~5–6s). Text output is exact.
  - --json exec returns the full native Claude JSON — session_id, result,
  total_cost_usd, full usage, modelUsage all present. No jq issues regardless of
   execution context (foreground or background).
  - Session inspection (summary, usage, session-id, session-file, tools,
  commands) all work correctly when you pass an explicit session ID.
  - Resume: same session ID returned, subagent retained prior context exactly.
  - --cwd flag works cleanly for cross-directory execution without changing
  shell state.

  Issues found:

  1. Session resolver picks the wrong session by default. When the main Claude
  Code conversation session and a subagent session share the same project
  directory, claude-result session-id (no args) returns the main session — which
   is continuously being touched. Running claude-result summary then shows my
  own response text, not the subagent's. The skill correctly warns about this
  ("For automation, prefer explicit session IDs"), but it's a footgun in
  practice. Any workflow that omits --json and tries to inspect with session-id
  afterward will silently get the wrong session.
  2. --allowedTools constraint is not enforced. When I passed --allowedTools 
  Glob, Claude still used Bash. The tool constraint doesn't reliably gate
  execution. Not a helper bug — a Claude runtime quirk — but the skill doesn't
  call this out.
  3. --json output format differs from copilot's. Claude returns the raw CLI
  JSON. model is not at .model — it's under
  .modelUsage["claude-sonnet-4-6[1m]"]. session_file is not included. Copilot
  normalizes these. Not wrong, but creates inconsistency if you try to use a
  unified pattern across skills.

  Codex-subagent

  Overall: Cleanest design. File-based paradigm sidesteps every session resolver
   problem.

  What worked:
  - Sync exec: clean, correct, fast enough.
  - summary, usage, session-id all work perfectly from the JSONL file.
  - commands extraction is excellent — gives cmd, exit, full out in one clean
  JSON object per command. Codex used /bin/zsh -lc 'ls -1A' and the extraction
  captured it perfectly.
  - Resume: works reliably. Session ID is read from the file, passed explicitly
  — no ambiguity. Resume correctly answered "10" with full context from the
  previous session.
  - No ANSI contamination in any context — because everything is redirected to a
   temp file first and the helper reads from disk. This is the right
  architecture.

  Issues found:

  1. No --json normalized wrapper. Unlike copilot, there's no single call that
  returns {session_id, result, model, usage} in one shot. You get the JSONL
  stream, then pipe to codex-result. In practice this means two commands for
  every sync workflow. The codex skill is explicit about this and the design is
  intentional, but it's more verbose.
  2. SKILL.md model claims will drift. It hardcodes defaults (gpt-5.4, high
  effort, fast tier) which is useful but will become stale. Worth a comment that
   these reflect ~/.codex/config.toml at time of writing.
  3. Async pattern warning is excellent and unique. The codex SKILL.md has a
  standout section warning against combining shell & with run_in_background on
  the Bash tool — that's a real pitfall and no other skill documents it. This
  should be considered for the others.

  ---
  Copilot-subagent

  Overall: Works in foreground. Broken in background contexts. Resume chain 
  snapped by ANSI contamination.

  What worked:
  - Foreground sync exec: works correctly, returns smoke-ok.
  - --json exec in foreground: returns clean normalized object {session_id, 
  result, model, usage, session_file}. Session file verified to exist on disk.
  This is the best normalized output format across all four skills.
  - summary: works correctly.
  - tools extraction: correctly identified report_intent and view calls.
  - session-file mode: resolves path correctly.
  - Model field works — returns gpt-5.4 correctly.

  Issues found:

  1. ANSI control character contamination in background execution. When copilot 
  exec runs in a Bash tool background invocation (PTY-allocated shell), the CLI
  emits ANSI progress/spinner output to stdout even though --output-format json
  is specified. The jq then fails to parse the result: jq: parse error: Invalid 
  string: control characters from U+0000 through U+001F must be escaped at line 
  4, column 10. This is a systematic failure — not a one-off. Every background
  --json exec call will be broken until this is stripped.
  2. Resume chain breaks when --json fails. Because session_id came back empty
  from the failed parse, the resume call used --resume "", which started a fresh
   session. Resume answered "I didn't list any items in this conversation — 0."
  instead of the correct count. So the failure cascade is: background → jq error
   → empty session ID → broken resume → wrong answer.
  3. Default model in SKILL.md is wrong. SKILL.md says gpt-5.3-codex but actual
  served model is gpt-5.4. Config drift.
  4. --json exec not safe for background Bash tool calls as documented. The
  "Asynchronous" pattern in the SKILL.md uses & inside the shell — that's the
  codex-only-approved pattern, and for copilot in a background Bash tool context
   it's broken. The skill needs either ANSI stripping (sed 
  's/\x1b\[[0-9;]*m//g') or a note that --json is foreground-only.

  Cursor-subagent

  Overall: Richest inspection data. Same ANSI background issue as copilot. 
  Incomplete task result observed.

  What worked:
  - Foreground sync exec: works. (Background exec produced ANSI contamination.)
  - summary: works — correctly shows MODEL: Composer 2 Fast, full usage, last
  message.
  - tools extraction: best in class. Shows every tool call with full input —
  Glob, Shell, Grep, all captured.
  - commands extraction: most detailed of any skill. Shows rejected vs success
  status, exit code, stdout, stderr, reason, is_readonly for rejected commands.
  A find command was rejected, and this was faithfully captured with status: 
  "rejected". A subsequent ls -1pa succeeded and its full output is in the
  record.
  - --trust is baked into every exec call — no need to pass it separately.
  - Stream JSONL saving alongside the transcript is a good design — the stream
  has richer data (tool call details) than the Cursor transcript alone.

  Issues found:

  1. Same ANSI contamination in background execution. jq: parse error: Invalid 
  string: control characters from U+0000 through U+001F must be escaped at line 
  20, column 10. Identical failure mode as copilot. Every background --json exec
   call will fail.
  2. --json output is missing model, session_file, stream_file. The --json flag
  returns just the raw result event: {type, subtype, is_error, duration_ms, 
  result, session_id, request_id, usage}. The SKILL.md documents this correctly
  ("print the final result event") but the quick reference says "Raw JSON
  result" — same terminology as claude, which is fine, just be aware you need
  separate calls to get model or file paths.
  3. Result field always has leading newline. "result": "\nsmoke-ok". The smoke
  test trims this with perl -0pe, but any caller who just does jq -r '.result'
  will get a leading newline. Should be stripped in the helper itself.
  4. Incomplete task result on file listing. Cursor listed only .DS_Store,
  .gitignore, LICENSE, README.md — missing all four subdirectories and scripts/.
   This was because the first Shell command (find ... -maxdepth 1 -type f) was
  rejected (despite --trust), then it ran multiple Globs that only found
  specific files, and the final ls -1pa (which DID succeed and showed
  directories) wasn't reflected in the final answer. The agent gave up
  constructing the list before incorporating the ls output. This is a cursor
  reasoning quality issue, not a helper bug.
  5. Resume broken via background. Session ID wasn't captured, resume started
  fresh session.
